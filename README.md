# FACEIT Form Level

Кроссбраузерное расширение (Manifest V3) для **FACEIT (CS2)**. Показывает
расширенную статистику игроков прямо в комнате матча и рассчитывает
независимый **Form Level** — «уровень формы» за последние матчи, в дополнение
к официальному уровню из ELO.

Источник данных — **только официальный FACEIT Data API**
(`open.faceit.com/data/v4`). Никаких трекеров и аналитики. API-ключ вводит сам
пользователь и хранится локально (`storage.local`).

---

## Возможности

Для каждого игрока обеих команд в лобби:

- Официальный уровень FACEIT + текущий ELO
- **Form Level** — второй уровень 1–10 по фактической игре за окно матчей
- Win rate за последние ~20 матчей
- Средний K/D и K/R
- HLTV-подобный рейтинг и средний ADR, размер выборки
- Бейдж вида `LVL 7 · Form 9 ▲▲` (стрелка = расхождение Form vs ELO-уровень)
- Тултип при наведении: rating, winrate, стабильность, размер выборки
- `<10` матчей → бейдж приглушён, пометка «мало данных»
- Нет данных по мультикиллам → пометка `~` (приблизительный расчёт)

## Сборка

```bash
npm install
npm run build      # → dist/chromium и dist/firefox
npm run watch      # пересборка при изменениях
npm run test       # юнит-тесты (vitest)
npm run typecheck  # tsc --noEmit
```

Стек: **TypeScript + esbuild**. Одна команда `npm run build` генерирует оба
артефакта:

- `dist/chromium/` — Chrome / Opera / Yandex (MV3, service worker)
- `dist/firefox/` — Firefox (MV3, background scripts, `browser.*`)

Общий код разделяется через `webextension-polyfill`.

> **Требуется Node.js 18+** (нужен глобальный `fetch`). На машине без Node
> установите его с https://nodejs.org, затем повторите команды выше.

## Как получить FACEIT API-ключ

1. Откройте <https://developers.faceit.com> и войдите под своим FACEIT-аккаунтом.
2. Создайте приложение (App).
3. В разделе **API Keys** создайте **Server-Side** ключ.
4. Скопируйте ключ и вставьте его на странице настроек расширения
   (кнопка «Проверить ключ» подтвердит, что он рабочий).

Ключ **не хардкодится** и хранится только в `storage.local` вашего браузера.

## Схема API

> ⚠️ **Самое хрупкое место проекта.** FACEIT Data API возвращает почти все
> числа **строками** (`"18"`, а не `18`), имена полей мультикиллов и раундов
> нестабильны, а в `/games/cs2/stats` некоторых полей может не быть вовсе.

### Как устроена защита

- `src/lib/api-schema.ts` содержит:
  - безопасные геттеры `num()`, `bool()`, `str()` — перебирают несколько
    возможных имён поля (алиасов) и возвращают нормализованное значение,
    **никогда не падая** (`"18" → 18`, `"" → 0`, `undefined → 0`);
  - таблицу `ALIASES` — соответствие «реальное имя поля API» → «наше поле»;
  - `roundsFromScore()` — вывод числа раундов из `Final Score` / `Score`
    (`"13 / 7" → 20`), когда поля `Rounds` нет.
- Флаг `hasMultikillData`: если поля мультикиллов недоступны, Form Level
  считается по одиночным фрагам (это занижает рейтинг), поэтому бейдж
  помечается `~`, а в тултипе появляется предупреждение.

### Реальные имена полей (проверено probe-скриптом, 2026-07-15)

Схема сверена живым ключом на регионе EU. Все поля ниже подтверждены в ответе
`GET /players/{id}/games/cs2/stats` (значения приходят строками):

| Наше поле | Реальное имя в API | Примечание |
|-----------|--------------------|------------|
| kills     | `Kills` | |
| deaths    | `Deaths` | |
| assists   | `Assists` | |
| rounds    | `Rounds` | присутствует всегда |
| adr       | `ADR` | |
| result    | `Result` (`"1"`/`"0"`) | результат самого игрока |
| double    | `Double Kills` | ✅ есть → Form Level точный, без `~` |
| triple    | `Triple Kills` | ✅ |
| quadro    | `Quadro Kills` | ✅ |
| penta     | `Penta Kills` | ✅ |
| elo       | `games.cs2.faceit_elo` | |
| level     | `games.cs2.skill_level` | |
| country   | `country` (`"ru"`) | флаг игрока |
| totalMatches | lifetime `Matches` | общее число игр |

**Нюанс счёта:** `Score` — это полный счёт карты (`"16 / 14"`), а `Final Score`
— одиночное число одной команды. Для вывода раундов из счёта берём `Score`
(алиас `finalScore` в `api-schema.ts` начинается со `Score`).

**Посегментная статистика (`segments`)** содержит те же поля по каждой карте
(`label: "Mirage"`, `Win Rate %`, `Average K/D Ratio`, `ADR`, `Matches`…) —
основа для будущей фичи «статы на текущей карте».

### Сверка реальной схемы (обязательный шаг)

```bash
# PowerShell
$env:FACEIT_API_KEY="ваш_ключ"; npm run probe -- s1mple
# bash
FACEIT_API_KEY=ваш_ключ npm run probe -- s1mple
```

`scripts/probe-api.ts` дёргает **player**, **player stats (cs2)** и
**`/games/cs2/stats?limit=20`**, печатает сырой JSON и **объединённый список
всех ключей** внутри объектов `stats`, а также проверяет наличие полей
мультикиллов и раундов.

### Что делать, если FACEIT поменял поля

1. Запустите `npm run probe` и посмотрите раздел «UNION of keys».
2. Добавьте новые имена в соответствующие массивы `ALIASES` в
   `src/lib/api-schema.ts` (порядок = приоритет).
3. `npm run test` и `npm run build`.

Логика расчёта имена полей **не** содержит — только `api-schema.ts`. Поэтому
адаптация к изменениям API сводится к правке одного файла.

## Калибровка Form Level

Модуль `src/lib/form-level.ts` реализует ровно 4 шага:

1. Рейтинг за каждый матч (HLTV 1.0-подобный).
2. Свёртка по окну матчей: среднее, взвешенное по раундам, с экспоненциальным
   затуханием по свежести (`recencyDecay`).
3. Поправка на винрейт: `Score = Rating · (1 + winrateWeight · (WinRate − 0.5))`.
4. Маппинг `Score → уровень 1–10` по массиву из 9 порогов.

Все параметры редактируются на странице настроек **без пересборки**:

| Параметр | Дефолт | Смысл |
|----------|--------|-------|
| `matchWindow`   | 20   | сколько последних матчей брать |
| `recencyDecay`  | 0.96 | `1.0` = все равны; `<1` = свежие важнее |
| `winrateWeight` | 0.20 | насколько винрейт двигает score |
| `thresholds`    | `[0.70, 0.80, 0.88, 0.95, 1.02, 1.09, 1.16, 1.25, 1.35]` | 9 границ между 10 уровнями |

> **Пороги — предварительная калибровка «на глаз», не эталон.** Их следует
> подгонять под реальное распределение вашей выборки игроков.

### Как подгонять пороги

1. Соберите Form Score (тултип показывает `rating`; сам `score` можно
   залогировать) для десятков игроков известного уровня.
2. Постройте распределение score по уровням.
3. Выставьте `thresholds` так, чтобы границы попадали между кластерами
   уровней (например, медиана между средним score LVL 6 и LVL 7).
4. Если данных по мультикиллам нет (`~`), score систематически занижен —
   калибруйте отдельно либо ориентируйтесь на относительный порядок игроков.

Кнопка «Сбросить к дефолтам» на странице настроек возвращает пороги и
параметры к значениям из `DEFAULT_FORM_CONFIG` (API-ключ сохраняется).

## Rate limits и кэш

- Все ответы кэшируются в `storage.local` с настраиваемым TTL (по умолчанию
  60 минут). Идентичность игрока и история матчей кэшируются отдельно.
- Запросы к игрокам лобби батчатся с ограничением параллелизма
  (`CONCURRENCY = 4` в `src/background/index.ts`).
- Form Level пересчитывается из кэшированной истории на каждый ререндер лобби
  **без** обращения к API; смена параметров формы применяется мгновенно.
- Ошибки (нет ключа / rate limit / игрок не найден) показываются понятным
  статусом на бейдже, расширение не падает.

## Приватность

Расширение обращается только к `open.faceit.com` (host permission) и работает
на страницах `www.faceit.com` (content script). Никакой телеметрии,
сторонних запросов и сбора данных.

## Замечания по надёжности DOM

FACEIT — это SPA, вёрстка меняется часто. В новом CS2-руме ники игроков —
это **не ссылки**, а `div` с классом-префиксом `Nickname__Name` внутри строки
`ListContentPlayer__Body`. Content script находит их по **стабильному префиксу
имени компонента** (`[class*="ListContentPlayer"] [class*="Nickname__Name"]`),
а не по полному хешированному классу styled-components, поэтому смена хеша при
деплое FACEIT его не ломает. Бейдж вставляется **после** ника
(`insertAdjacentElement('afterend', …)`), не трогая разметку FACEIT.
Отслеживание динамической подгрузки — через `MutationObserver` + перехват
`history.pushState/replaceState`.

Если FACEIT переименует компоненты, поправьте `ROSTER_NAME_SELECTOR` в
`src/content/index.ts` (снимите актуальную структуру диагностическим сниппетом
из истории или через DevTools → Inspect).
